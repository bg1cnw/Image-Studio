# 比例 Auto 与分辨率的关系

这份文档专门解释一个常见疑问:为什么把「比例」切到 `Auto` 以后，分辨率也不能再像独立开关一样单独锁定。

## 先说结论

- Image Studio 底层保存和发送的是一个统一的 `size` 字段，不是分别发送“比例”和“分辨率”两个独立参数。
- 选择具体比例和具体分辨率时，应用会先把两者组合成一个明确的像素尺寸，例如 `1536x1024`、`2048x1152`、`2160x3840`。
- 当你显式选择 `Auto` 时，底层 `size` 会直接变成 `"auto"`，意思是“把最终尺寸交给上游决定”。
- 一旦 `size` 已经是 `"auto"`，应用本地就不再持有一个可继续锁定的明确像素尺寸，所以“分辨率”也不会再保留成一个独立的固定值。

## 为什么看起来像“比例自动后，分辨率失控了”

界面上有“比例”和“分辨率”两个选择区，但它们只是同一个 `size` 的两种拆分视图，不是两条彼此独立的请求字段。

可以把它理解成这样:

- `1:1 + 1K` -> `1024x1024`
- `3:2 + 1K` -> `1536x1024`
- `16:9 + 2K` -> `2048x1152`
- `9:16 + 4K` -> `2160x3840`
- `Auto` -> `"auto"`

也就是说，`Auto` 不是“只自动比例，但仍然固定分辨率”，而是“整个尺寸都交给上游”。

这也是为什么当前实现里，只要 `size === "auto"`:

- 当前激活比例会显示为 `Auto`
- 当前激活分辨率也会显示为 `Auto`

不是界面忘了记住你之前的分辨率，而是底层已经没有一个同时满足“仍是 Auto”又“还固定 2K / 4K”的单一 `size` 值可发给上游。

## 两种 API 形态都是同一个语义

无论当前走的是:

- Responses API
- Images API

最终发给上游的尺寸都是单个 `size` 值。

所以这里不是某个平台面板的特殊行为，也不是桌面端和 Android 端语义不一致，而是整个应用共享的尺寸模型就是这样设计的。

## 如果想重新拿回分辨率控制，应该怎么做

只要你点一个明确的比例，或点一个明确的分辨率，应用就会把状态从 `auto` 提升回一个具体像素尺寸。

当前规则是:

- 从 `Auto` 状态点击具体比例:
  - 如果分辨率这边还是 `Auto`，应用会先用 `1K` 作为默认档位再生成具体尺寸。
  - 例如点击 `9:16`，会得到 `864x1536`。
- 从 `Auto` 状态点击具体分辨率:
  - 如果比例这边还是 `Auto`，应用会先用 `1:1` 作为默认比例再生成具体尺寸。
  - 例如点击 `2K`，会得到 `2048x2048`。
- 只有当你再次明确点击 `Auto`，才会重新回到“由上游决定尺寸”的状态。

## 哪些情况是正常语义，哪些情况才算 bug

正常语义:

- 你主动选择 `Auto` 后，分辨率不再作为独立锁定值保留。
- 不同模型能提供的比例和分辨率组合不同。
- 某些链路只提供基础尺寸，不提供 `2K` / `4K` / 自定义比例。

这类情况通常是模型能力或链路能力限制，不是交互故障。

真正算 bug 的情况是:

- 你当前在 `Auto`
- 你已经点击了一个明确比例或明确分辨率
- 结果状态仍然卡在 `Auto`，没有退出

这个“点了具体项却仍卡在 Auto”的互锁问题已经在 2026-05-30 修过；当前桌面端和 Android 端都应该在点击明确项后正常退出 `Auto`。

## 使用建议

- 想让上游自己决定最终宽高，就用 `Auto`，同时接受“本地不再锁定具体分辨率”。
- 想稳定控制输出像素尺寸，就不要停留在 `Auto`，而是至少选择一个明确比例或明确分辨率。
- 如果你需要 `2K`、`4K`、自定义比例这类能力，先确认当前模型和请求策略本身支持这些选项。
